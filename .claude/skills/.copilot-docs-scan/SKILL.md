# 仕様書のバグ発見

`.docs/` ディレクトリの仕様書を読んで、仕様の問題や矛盾を発見する。

## 方針

- 仕様書の整合性をチェック
- コードとの乖離を発見
- ユーザー目線で矛盾や不明点を指摘
- 見つけた問題を報告する

## 点検範囲

### 仕様書の整合性

- 仕様書間の矛盾（features, services, models の不整合）
- 用語の不統一（glossary との乖離）
- 古い情報（実装と異なる仕様）
- 不明確な記述（複数の解釈が可能）

### コードとの整合性

- 実装済みだが仕様書に反映されていない機能
- 仕様書にあるが実装されていない機能
- 仕様と異なる実装

### ユーザー体験

- 理解しにくい説明
- 欠けている情報
- ワークフローの矛盾

## プロセス

### 仕様書の構造理解

`docs` スキルで仕様書の構造とルールを理解する。

### 仕様書の調査

`.docs/` ディレクトリを体系的に調査:

1. `overview.md` - 製品概要の整合性
2. `features/` - 機能仕様の整合性
3. `services/` - サービス仕様の整合性
4. `models/` - データモデルの整合性
5. `glossary.md` - 用語の統一性
6. `architecture.md` - アーキテクチャとの整合性

### コードとの突き合わせ

Glob と Grep を使って関連コードを確認:

- 仕様書に記載された機能が実装されているか
- 実装されている機能が仕様書に記載されているか
- 仕様と実装が一致しているか

### 問題の分類

発見した問題を分類:

**即座に修正（自律的に対応）:**
- 明らかな typo
- フォーマットの不統一
- リンク切れ

**報告（人間の確認が必要）:**
- 仕様の矛盾
- コードとの乖離
- 不明確な記述
- 欠けている情報

### 問題への対応

#### 自律的に対応

軽微な問題は即座に修正してコミット。

#### 人間の確認が必要な問題

報告に含める内容:

- 問題の内容（何が問題か）
- 発見箇所（どのファイル・セクション）
- 影響範囲（どのユーザー・機能に影響するか）
- 推奨する対応（どう修正すべきか）
- 推奨優先度（高/中/低）

発見した問題を成果物として提出して終了する。修正は行わない（typo/フォーマット等の自律的な修正を除く）。
