# stories/

業務ストーリー。ロール横断のユースケースを「実際の現場で何が起きるか」の物語として書く。`user-flows.md` が1ロール1導線の番号付きステップなのに対し、stories は複数ロールと時間が絡む業務の流れを散文で描く。

## いつ作るか

- 業務の流れが複数ロールにまたがる（現場入力 → 承認 → 請求 → 経営確認 など）
- 機能単体では伝わらない「業務の全景」を新規参画者に渡したい
- 営業・社内説明で「使うとどうなるか」を物語で語る素材が必要
- features と user-flows で粒度が合わない（features は機能仕様、user-flows は1ロール1導線）

業務が単純で1ロール1画面で完結するなら不要。

## ディレクトリ構成

```
stories/
  index.md                    目次
  {業務名（日本語）}.md         1ストーリー1ファイル
```

ファイル名は業務の動詞句で書く。スペースはアンダースコアに置換する。例。

- `サービスを実施して記録する.md`
- `計画書をつくり提供するサービスを決める.md`
- `月次支出を取り込む.md`
- `月次資料をつくって予実を見る.md`
- `実績を請求につなげる.md`
- `年間予算をつくる.md`

slug 規則（英字ハイフン）で揃えたい場合は frontmatter `slug` を立て、ファイル名は英字に揃える。プロジェクトでどちらかに統一する。

## ルール

- 散文で書く。番号付きステップは使わない（それは user-flows.md の役割）
- 主語は実際のユーザー（ペルソナ）。「施設長は…」「介護職員は…」と書く
- 時間軸で章立てる（朝・午前・午後・夕方、または月初・月中・月末）。業務のリズムが伝わる
- 業務上の数字・固有名詞・道具（紙、PC、スマホ、IC タグ など）を具体に書く
- 1ストーリーあたり 20〜60 行を目安に。短すぎると物語にならず、長すぎると読まれない
- 「つながる機能」「関連する痛みと価値」セクションで features / signals / value にリンクする。stories はナレッジグラフのハブの1つになる
- ストーリーが想定通りに動かない場合（例外、エラー処理）は別ストーリーに分ける。1ファイルに分岐を盛らない

## index.md

業務ストーリーの目次。

```md
# 業務ストーリー

製品を使うと現場で何が起きるかを業務単位の物語で描く。機能単体は [[features/index|features]] を、1ロール1導線は [[user-flows]] を参照。

## {業務領域}

- [[{ストーリー名}]] — {1行サマリ}
- [[{ストーリー名}]] — {1行サマリ}

## {次の業務領域}

...
```

業務領域で分類する（記録・請求・経営・計画 など）。製品の業務単位に合わせて自然に立ち上がる分類を使う。

## user-flows.md との棲み分け

user-flows.md は1ロールが1つの導線をたどる番号付きステップ。E2E テスト生成のソース。

stories/ は複数ロール・複数日にまたがる業務の物語。営業・新人説明・痛みの可視化のソース。

両者は同じ製品の使われ方を別の粒度で記述する二輪。一方を他方に統合しない。
