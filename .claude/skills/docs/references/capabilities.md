# capabilities.md

ロール別「できること」サマリ。商談・営業・社内説明用の元情報として、実装済み機能のみをロール目線でまとめる。

`features.md` が「機能の責務」で並ぶのに対し、`capabilities.md` は「そのロールの人が何を得られるか」で並ぶ。営業資料の元ネタやオンボーディング説明に直接転用できる粒度。

## フォーマット

```md
# できること（ロール別サマリ）

{商談・営業・社内説明用の元情報。実装済み機能のみをロール別の「できること」目線でまとめる。フェーズ運用の注記}

機能単位の詳細は [[features/index|features]] を、未実装項目は [[backlogs/index|backlog]] を参照。

## {製品名} が解決すること

- {1行で書ける成果}
- {1行で書ける成果}

## {ロール名} ができること

- {動詞で始まる1行サマリ}
- {動詞で始まる1行サマリ}

## {次のロール名} ができること

...
```

## ルール

- 1行は動詞で始める（「確認」「比較」「入力」「出力」など）
- 業務上の数字・固有名詞は具体に書く（「9施設」「月別・施設別」「予算対比・前年比・前月比」）
- 機能名そのものでなく、機能を使って「何が達成できるか」を書く
- フェーズ運用がある製品では行末に `（Phase 2）` のような注記を付け、現フェーズで使えるものとそうでないものを区別する
- 未実装機能は書かない。書きたくなったら `backlogs/` へ
- 機能との対応関係を保つため、行末や別欄に `[[features/{slug}]]` を添えてもよい

## 維持の仕方

`features.md` または `features/{slug}.md` が更新されたとき、ロール別の見え方が変わる行だけ追従する。全体再生成はしない。商談・営業の人がこのファイルだけ読めば仕様の最新像が掴めるよう、温度感を持って更新する。
