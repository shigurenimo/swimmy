# value.md

製品が現場に対して何の痛みを解決し、どんな価値を生むかを記す。index.md の「解決する問題」が箇条書きの薄い入口なのに対し、value.md は現場の声と現状観察に基づく深掘り。

## いつ作るか

- 現場の痛みが多面的で、index.md の数行では伝わらない
- signals/ に集まった声を分析して「製品の核」を言語化したい
- 商談・社内説明で「なぜこの製品か」を語る元ネタが必要
- 外販を見据えて差別化の核を明文化したい

製品が単機能なら index.md だけで足りるので作らない。

## フォーマット

```md
# 提供価値

{1段落のリード。事実としての連携構造は [[integrations]]、データ構造は [[domain]] を参照、などの誘導}

## 解決する痛み

{現場の現状を散文で記述。複数ソフトの分断、二重入力、属人化、紙運用などを具体に書く。signals/ の該当 [[テーマ]] へリンクし、声の出所を辿れるようにする}

## {製品名} が生む価値

{解決後の状態を散文で記述。「一本でつなぐ」「単一の基盤になる」など、何が統合されて何が消えるかを書く}

{差別化の核。なぜこの製品がそれを実現できるか、他では実現しにくいかを1段落}

## 想定する成果

- {定量・定性の成果}
- {定量・定性の成果}
```

## ルール

- 散文で書く。箇条書きを羅列しない（痛みは文脈とセットで伝わる）
- 痛みは現場の具体に落とす。固有名詞、運用、数字を入れる（「9施設」「定期巡回」「ケアパレットへ転記」など）
- signals/ のテーマファイルから集約する。声と痛みは1対1ではなく、複数の声から1つの痛みが浮かぶ。脚注ではなく本文リンクで `[[signals/{slug}|声のテーマ名]]` を埋め込む
- 「製品が生む価値」は機能の列挙にしない。状態の変化を書く
- 外部連携の事実関係は書かない（[integrations.md](integrations.md) に置く）。価値の根拠として触れるだけ
- index.md と内容を重複させない。index.md は1段落のリード、value.md は本体

## 維持

signals/ の傾向が変わった、新しい痛みが浮上した、コードで新機能が立ち上がった、商談で語り方が固まった、というタイミングで更新する。月次・四半期で見直すと鈍化を防げる。
