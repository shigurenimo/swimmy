# glossary.md

ドメイン用語の定義。コードの命名と一致させる。製品固有用語と機能名の索引もここで兼ねる。

```md
## {日本語名} - {PascalCase}

{1-2文の説明}
```

## ルール

- 見出しは `## 日本語名 - PascalCase`
- 本文は1-2文で説明
- コードの命名と一致させる。不一致があればコード側を修正
- 機能詳細との二重記述を避け、製品の用語はここに留めて細かく書かない

## references/terms

製品の外にある専門知識（業界用語・制度・会計・技術）は `references/terms/{用語}.md` に1用語1ファイルのアトミックなノードとして定義する。`glossary.md` はそれらの索引も兼ねる（terms 配下に index.md は作らない）。

ノードのフォーマット。

```md
---
term: 用語
reading: よみがな
aliases: [別名]
category: カテゴリ
---

# 用語（正式名）

2〜6 行の定義。素人にも分かる言葉で。関連語や深掘り先の notes は、定義文の中に自然に [[]] で埋め込む。
```

ルール。

- 1ファイル1用語。定義は短く（2〜6行）
- リンクは定義文の中に自然に `[[]]` で埋め込む。「関連」「詳細」のようなリンク専用の節やリストは作らない（リストの維持がメンテ負担になる）。深掘り先の `notes/` も文中でリンクする
- 製品固有の挙動・機能はノードにせず、該当機能 `[[features/{slug}|表示名]]` へリンクする
- ファイル名に大文字を使わない。英字の用語は小文字とハイフンだけ（例: `pl.md`、`kpi.md`、`spo2.md`）。frontmatter `term:` と本文見出しには `PL`・`SpO2` のような正式表記を残し、リンクは `[[pl|PL]]` で表示名を当てる。日本語の用語は日本語ファイル名でよい
- 外部 SaaS や周辺製品の詳細調査は `references/{system-name}/` でサブツリー化してよい（例: `references/smacare/`）。terms はあくまで短い定義に留め、長い解説はそちらへ
