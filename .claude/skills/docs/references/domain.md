# domain.md

ドメインモデルを業務語で整理する。コードを正のソースとし、スキーマファイル（例: `src/schema.ts`）から抽出する。

## いつ作るか

- ドメインが複雑（テーブル数20以上、エンティティ間の関係が業務知識を要する）
- 業務担当者（経理・営業・現場）がデータ構造を理解する必要がある
- features/ や user-flows.md だけでは「何と何が紐づくか」が掴めない

ドメインが単純なら作らない。

## フォーマット

```md
# データモデル

{製品名} が扱うデータを業務語で整理する。{現場入力 → 請求 → 経営など、価値の流れ}を一本でつなぐ構造。

このドキュメントはコードを正とする。源泉は `{schema ファイルパス}`（現状 N テーブル）。

## エンティティ一覧

カテゴリ別に並べる。括弧内は業務上の意味。

{カテゴリ名}

- {テーブル名}（{業務上の意味}）
- {テーブル名}（{業務上の意味}）/ {関連テーブル名}（{意味}）

{次のカテゴリ名}

- ...

## 主要な関係

{業務上重要なエンティティ間の関係を mermaid erDiagram で}

## ライフサイクル

{エンティティが業務上どう生まれ・更新され・消えるか。1〜3エンティティ}
```

## ルール

- 業務語を主語に書く（テーブル名は補助）。「実装詳細」ではなく「業務上何を表すか」
- カテゴリは業務領域で分ける（認証・組織、職員、利用者、記録、請求、会計 など）
- 主要なエンティティ間の関係は mermaid `erDiagram` で1枚
- 属性は書かない（コードが仕様）。業務上のキー属性が必要な場合のみ短く触れる
- 用語は glossary.md と一致させる。terms（業界知識）には `[[]]` でリンク
- スキーマファイルの場所とテーブル数を冒頭で宣言。テーブル追加時はこの数も更新
- 外部システムとのデータやりとりは `integrations.md` を別途立てる。domain.md は内部の業務データ構造に集中する
