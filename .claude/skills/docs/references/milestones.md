# milestones.md

リリース計画。複数フェーズがある製品でのみ作る。各機能ファイルの frontmatter `milestone:` をこのファイルの日付に揃えることで、機能とリリースを双方向に辿れる。

## フォーマット

```md
# マイルストーン

{戦略の要約。フェーズ分けの背景・出典（議事録）への wikiリンク}

## YYYY-MM-DD — {フェーズ名}

目的: {このマイルストーンで何を達成するか}

### 含む機能（N件）

{ナラティブ説明。戦略の核と、どの機能が中核か}

#### 実装ステータス一覧

いつまでに・なにが・どこまで実装されるかの俯瞰。最終更新 YYYY-MM-DD、実装済 X・一部 Y・未着手 Z。

{ロール別または責務別に [[features/{slug}|表示名]] を列挙し、各行末に（実装済 / 一部 / 未着手）を付ける}

### 未着手の主な課題

- {残っている検討項目}
- {外部依存}

## YYYY-MM-DD — {次のフェーズ名}

...

## 完了済み

過去のマイルストーンも履歴として残す。日付セクションをそのまま下に積む。
```

## ルール

- 機能は `[[features/{slug}]]` 形式の wikiリンクで参照する
- 機能ファイルの frontmatter `milestone:` の値はこのファイルに登場する日付のいずれかと一致させる
- 優先度（must / nice）は使わない。線はマイルストーン日付だけで引く。あるマイルストーンに含まれること自体がそのフェーズのマストの意味になる。「あればよい」相当は後ろのマイルストーンへ送る
- フェーズ分けの根拠は議事録に置き、`[[sources/minutes/YYYY-MM-DD-{topic}]]` で逆リンクする
- マイルストーン途中で機能が外れたら削除せず、Phase X へ降格などの注記を残す（決定の履歴になる）
- 完了済みマイルストーンも消さない。履歴として残す

## frontmatter 連携

機能ファイル（[features.md](features.md) 参照）。

```yaml
---
name: 月次損益分析
slug: monthly-pl
milestone: 2026-07-01
---
```

`milestone:` が `milestones.md` の日付セクションと不一致になったら検証エラー（[validation.md](validation.md) を参照）。
