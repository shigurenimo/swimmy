# roles-and-permissions.md

ロールごとのアクセス可能エンドポイント・画面・操作を、コードから抽出して整理する。実装が正のソース。変更時はこのファイルも更新する。

## いつ作るか

- ロールが2つ以上ある
- ロールごとにアクセス境界が分かれている（管理 / 経理 / 現場 など）
- 認可ルールがコードに散らばっていて俯瞰が必要

ロールが1種類しかなければ作らない。

## フォーマット

```md
# ロールと権限

このドキュメントは {抽出元、例: `src/api/interface/routes/` 配下の `requireRole(...)`} で宣言されている認可ルールを抽出したものです。実装が正のソースであり、変更時はこのドキュメントも更新してください。

## ロール一覧

- {ロール名}（{業務上の役割}）: {スコープの説明（全社横断 / 自施設限定 など）}
- {ロール名}（{業務上の役割}）: {スコープの説明}

## 設計方針

- {横断スコープのロール} は全 {単位} を参照・操作できる
- {限定スコープのロール} は自分の所属 {単位} のみ
- {認可ヘルパ名} で許可ロールを宣言する規約。引数なしで呼ぶと全ロール 403 になる挙動の意味
- ハンドラ内のクロスチェックで {限定スコープのロール} を自分の {単位} に縛る

## {ロール名} のみアクセス可能なエンドポイント

- {path}（{機能名}）
- {path}（{機能名}）

## {次のロール} のみアクセス可能なエンドポイント

...

## 共通アクセス

...
```

## ルール

- 冒頭に抽出元を明記し、「実装が正」を宣言する
- ロール名は業務上の役割（経営、経理、施設長、介護職員 など）を添える
- スコープ（横断 / 限定）を必ず書く。これが認可設計の中核
- エンドポイントは認可ヘルパの引数を grep して機械的に抽出する。手で並べ替えない
- 一時的にロックされているエンドポイント（意図的な封鎖）はその旨を明記する
- 廃止ロールはセクションごと残し、「廃止」とだけ書く（履歴になる）

## 維持

実装変更時の検証。

- 新しいエンドポイントが追加された
- 既存エンドポイントの `requireRole(...)` 引数が変わった
- ロールが追加・廃止された

これらをトリガに該当セクションを更新する。コード側のヘルパ名や宣言方法が変わったら冒頭の抽出元宣言も合わせて直す。
