# api-schema.md

API がある場合のみ作成する。エンドポイント一覧ではなく、構造を伝える。

## いつ作るか

- 外部に公開する API がある
- 複数フロントエンド（Web・モバイル）が同じ API を使う
- 認証方式や規約を一箇所に集めたい

内部用の Server Actions だけなら不要（コードと型を読めば十分）。

## フォーマット

```md
# API Schema

## 認証

{方式（JWT、OAuth、API Key など）}
{ヘッダ・有効期限・更新方法}

## エンドポイント構造

- ベース URL: {origin}
- バージョニング: {方式}
- ルート命名: {規約}
- レスポンス形式: {JSON / GraphQL など}

## エラー

- ステータスコードの方針
- エラーレスポンスの形

## レート制限

- {あれば}
```

GraphQL の場合。

- スキーマの主要な型とクエリ/ミューテーションの概要
- 個別フィールドはコード（schema.graphql など）が source of truth

## ルール

- 認証方式とエンドポイントの構造を書く
- 個別エンドポイントの一覧は書かない（コードと OpenAPI / GraphQL スキーマで足りる）
- 外部公開しているなら API リファレンス（自動生成）への入口リンクを最後に置く
