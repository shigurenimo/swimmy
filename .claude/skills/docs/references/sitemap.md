# sitemap.md

全ページを箇条書きで列挙する。サイドバーで実際に触るロールごとに並べる視点で書く。

## 分類軸の使い分け

`features/index.md` は「機能の責務がどのロールの業務領域に属するか」で並ぶ。`sitemap.md` は「サイドバーに実際に出る URL」をロールごとに表示順で並べる。

両者は同じ画面を別の軸で索引する。たとえば `/invites` `/audit-logs` のような組織横断の土台機能は、`features/index.md` では「全ロール共通」に置き、`sitemap.md` では実際に触る「経営者」セクションに置く。違いは設計通り。

## フォーマット

```md
# Sitemap

各ロールでログインしたときにサイドバーに出る URL を、表示順に列挙する。

機能の責務単位での分類は [[features/index|features]] を参照。

セクションの状態区分。

- 区分なし: {現フェーズ}マスト確定
- 「{次フェーズ} 要相談」: {現フェーズ}内で組み込めるか要件詰め次第

## Public

- /register - {説明}

## Auth Required

- /settings - {説明}
- /profile - {説明}

## {ロール名}

- /{path} - {説明}
- /{path} - {説明}（補足: {Phase 2 降格などの注記}）

## {次のロール名}

...

## 概念パス（実 URL なし）

- /_overview/{path} - {説明}

## 廃止・リダイレクト

- /old-path → /new-path
```

## ルール

- セクション分けはアクセスロールで行う（Public, Auth Required, ロール別）
- 形式は ``- /path - 説明``
- 動的パラメータは `:param` 表記
- 多言語バリアントは本体と別に書かない。注記で「全公開ページに /en/ あり」とだけ書く
- フェーズ降格・サイドバー非表示にした機能は、本ファイルから外す。実装・ルートが残っている場合は冒頭で「降格機能は記載しない。実装は維持」と一言入れる
- 概念スライド用の `/_overview/...` パスは実 URL ではないことを明記する
- 廃止パスとリダイレクトは末尾の別セクション
