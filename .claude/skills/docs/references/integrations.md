# integrations.md

製品が現場で併用する外部システムや公的機関と、何をどうやりとりするかを事実として整理する。なぜ統合するのかという価値・痛みは [value.md](value.md)、内部のデータ構造は [domain.md](domain.md)。

## いつ作るか

- 外部システム連携が2つ以上ある（SaaS、官公庁、業界標準フォーマット）
- 連携の役割分担を社内・社外に説明する必要がある
- 移行・置換の段階的シナリオを設計したい

連携が単発の OAuth ログインだけなら不要。architecture.md の Authentication 節で足りる。

## フォーマット

```md
# 外部システム連携

{1段落のリード。なぜ統合するかは [[value]] を参照、内部構造は [[domain]] を参照}

## 周辺の外部システム

{連携先を散文で列挙。各システムを1〜2文で紹介する。提供会社名・カテゴリ・特徴を含める}

## 現状のデータの流れ

{記録系・請求系・人事系などのカテゴリごとに、データが今どう流れているかを散文で記述}

{役割分担: どこまでが自製品、どこから外部システム、という境界を明示}

## やりとりされるデータ形式

{業界の標準フォーマットが定まっているもの、CSV や手入力に頼らざるを得ないもの、それぞれを分けて記述}

## 移行・置換のシナリオ

{段階的に置換する予定があるなら、フェーズごとに「何を引き取り何を残すか」を記述。なければセクションごと省略可}
```

## ルール

- 散文で書く。連携先と流れは文脈とセットで伝わる
- 連携先システムは初出時に提供会社・カテゴリ・特徴を1文添える（読者は介護業界・経理業界などのドメインに詳しいとは限らない）
- 役割分担を必ず明示する。「どこまでが自製品の責務か」を曖昧にしない
- 標準フォーマット（業界共通の様式・公的様式）が定まっているものと、ベンダーごとに様式が違うものを区別する
- 外部システム名は terms にノードを置き、本文では `[[smacare|スマケア]]` のようにリンクする。長い解説は `references/{system-name}/` のサブツリーへ
- 連携の現状と理想は分けて書く。現状は事実、理想は migration セクション
- API スキーマやエンドポイントの詳細は [api-schema.md](api-schema.md) に。integrations.md は業務上の流れに集中する
