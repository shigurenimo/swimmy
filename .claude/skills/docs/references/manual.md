# manual/

エンドユーザー向けのマニュアル。`features/` が「機能の責務と仕様」を網羅するのに対し、`manual/` は「現場の人がその手順をこなすためのガイド」。読者はロール別に分かれる。

## いつ作るか

- エンドユーザーが社内・現場の人で、機能名から手順を逆引きする必要がある
- 営業・サポート対応で「どう操作するか」の説明が頻繁に必要
- 新人オンボーディングの教材として使う

開発者だけが使うシステムでは作らない。

## ディレクトリ構成

```
manual/
  index.md                目次
  common.md               全ロール共通の前提（ログイン、画面構成、共通操作）
  {role}.md               ロール別マニュアル（経営、経理、施設長、介護職員 など）
```

ロールが多い・厚い場合は `{role}/` ディレクトリに分けて、章ごとにファイルを切ってもよい。

## フォーマット

```md
# {ロール名} マニュアル

このマニュアルは {ロール名} が日々の業務でシステムを使いこなすための手順書です。

## 1日の流れ

朝。

- ログインして {ホーム画面名} を開く
- {確認する項目}

午前。

- {作業}

午後。

- {作業}

夕方。

- {確認・登録}

## よくある操作

### {操作名}

1. {手順}
2. {手順}
3. {手順}

### {操作名}

...

## 困ったとき

- {状況}: {対処}
- {状況}: {対処}
```

## ルール

- ユーザーの言葉で書く。機能名や画面名は「ユーザーがどう呼んでいるか」に揃える
- 1日・1週・1月のリズムで章立てると現場の使い方が伝わる
- 操作手順は番号付きステップ。`user-flows.md` よりも具体（クリック先・入力値の例まで）
- スクリーンショットを置くなら `manual/images/` に置き、本文からファイル名で参照する
- 困ったときセクションを必ず置く。サポート問い合わせの一次対応を吸収する
- features/ の機能仕様と矛盾しないよう、用語と操作は整合させる。manual/ の更新でずれを発見したら features/ も直す

## features との棲み分け

features/ は機能を主語に、仕様と境界を、開発者・営業・PM 向けに、仕様変更時に更新する。

manual/ はユーザーを主語に、手順とコツを、エンドユーザー向けに、運用フィードバック時に更新する。

両者は同じ機能を別の視点で記述する。一方が source of truth ではなく、整合性を保つ二輪。manual/ の更新でずれを発見したら features/ も直す。
